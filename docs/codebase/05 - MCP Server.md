# MCP Server

`mcp/memory_server.py` exposes the companion's memory and capabilities as MCP tools. The protocol is JSON-RPC 2.0 over stdio, meaning the server reads JSON requests from stdin and writes JSON responses to stdout. This is the standard MCP transport, designed for subprocess communication where the Claude CLI spawns the server and manages its lifecycle. The memory server is the primary way the agent interacts with its persistent state - every memory write, observation, thread update, and Telegram message flows through these tools.

## Configuration

The MCP servers are launched as subprocesses by the `claude` CLI, not by the Electron main process directly. Configuration is generated by `src/main/inference.ts` and written to `mcp/config.json` at runtime. The Claude CLI reads this config file and spawns each server as a child process with the specified command, arguments, and environment variables.

The following JSON shows the structure of the generated config. Each server entry specifies a Python interpreter, the script path, and environment variables that the server reads on startup to locate its resources.

```json
{
  "mcpServers": {
    "memory": {
      "command": "/path/to/python",
      "args": ["mcp/memory_server.py"],
      "env": {
        "COMPANION_DB": "~/.atrophy/agents/<name>/data/memory.db",
        "OBSIDIAN_VAULT": "<vault-path>",
        "OBSIDIAN_AGENT_DIR": "<vault>/<agent_display_name>",
        "OBSIDIAN_AGENT_NOTES": "<vault>/<agent>/agents/<name>",
        "AGENT": "<name>"
      }
    },
    "puppeteer": {
      "command": "/path/to/python",
      "args": ["mcp/puppeteer_proxy.py"],
      "env": {
        "PUPPETEER_LAUNCH_OPTIONS": "{\"headless\": true}"
      }
    },
    "fal": {
      "command": "npx",
      "args": ["-y", "fal-ai-mcp-server@latest"],
      "env": {
        "FAL_KEY": "<fal-api-key>"
      }
    }
  }
}
```

### MCP Config Generation

The `getMcpConfigPath()` function in `src/main/inference.ts` handles config generation on every inference call. The config is cached after the first generation and only regenerated when `resetMcpConfig()` is called (which happens during agent switching). The generation process follows these steps:

1. Builds the server map with `memory`, `puppeteer`, and optionally `google` servers
2. The Google server (`mcp/google_server.py`) is added dynamically when `GOOGLE_CONFIGURED` is true
3. Imports global MCP servers from `~/.claude/settings.json` (any servers not already defined are merged in, allowing users to bring their own MCP servers)
4. Writes the config to `mcp/config.json` and caches the path for subsequent calls
5. The cached config can be reset via `resetMcpConfig()` (needed when switching agents, since each agent has different database paths and Obsidian directories)

The following TypeScript shows the structure built in code. Note that all paths come from the Config singleton, which resolves them based on the current agent and environment.

```typescript
const mcpConfig = {
  mcpServers: {
    memory: {
      command: config.PYTHON_PATH,
      args: [config.MCP_SERVER_SCRIPT],
      env: {
        COMPANION_DB: config.DB_PATH,
        OBSIDIAN_VAULT: config.OBSIDIAN_VAULT,
        OBSIDIAN_AGENT_DIR: config.OBSIDIAN_AGENT_DIR,
        OBSIDIAN_AGENT_NOTES: config.OBSIDIAN_AGENT_NOTES,
        AGENT: config.AGENT_NAME,
      },
    },
    puppeteer: {
      command: config.PYTHON_PATH,
      args: [path.join(config.MCP_DIR, 'puppeteer_proxy.py')],
      env: {
        PUPPETEER_LAUNCH_OPTIONS: JSON.stringify({ headless: true }),
      },
    },
    // google: added conditionally if config.GOOGLE_CONFIGURED
    // global servers: merged from ~/.claude/settings.json
  },
};
```

### Environment Variables Passed to Memory Server

The memory server reads these environment variables at startup to locate the agent's database and Obsidian vault. All paths are fully resolved by the Electron config module before being passed to the server.

| Variable | Source | Purpose |
|----------|--------|---------|
| `COMPANION_DB` | `config.DB_PATH` | Full path to the agent's SQLite database |
| `OBSIDIAN_VAULT` | `config.OBSIDIAN_VAULT` | Root path of the Obsidian vault |
| `OBSIDIAN_AGENT_DIR` | `config.OBSIDIAN_AGENT_DIR` | Agent's directory in the vault (for workspace operations) |
| `OBSIDIAN_AGENT_NOTES` | `config.OBSIDIAN_AGENT_NOTES` | Agent's notes directory (for read/write operations) |
| `AGENT` | `config.AGENT_NAME` | Agent slug (e.g. `xan`, `general_montgomery`) |

### Environment Variables Passed to Google Server

The Google server authenticates using OAuth credentials from the environment. These are loaded from `~/.atrophy/.env` or the shell environment by the Electron config module.

| Variable | Source | Purpose |
|----------|--------|---------|
| `GOOGLE_OAUTH_CLIENT_ID` | From `~/.atrophy/.env` or shell env | OAuth2 client ID for gws CLI |
| `GOOGLE_OAUTH_CLIENT_SECRET` | From `~/.atrophy/.env` or shell env | OAuth2 client secret for gws CLI |

### Python Path Resolution

The MCP servers require a Python 3 interpreter. The config module resolves the Python path using a cascade that tries multiple locations, since the Python installation path varies across macOS setups (Homebrew on Apple Silicon vs Intel, system Python, virtual environments).

1. `PYTHON_PATH` environment variable (highest priority - user override)
2. `python3` (system lookup via `execSync`)
3. `/opt/homebrew/bin/python3` (Homebrew on Apple Silicon)
4. `/usr/local/bin/python3` (Homebrew on Intel, or manual install)

If all fail, falls back to the string `'python3'`, which will cause a clear error if Python is not on the PATH.

### Bundling

MCP servers are bundled as `extraResources` in the Electron app via `electron-builder.yml`. This means the Python scripts are copied into the application bundle at build time, alongside the whisper binary, agent definitions, and database schema.

```yaml
mac:
  extraResources:
    - from: mcp/
      to: mcp
```

At runtime, the MCP server paths are resolved relative to the bundle root (`process.resourcesPath` when packaged, project root in development). This dual-path resolution ensures the same code works in both development and production without path changes.

### Server Summary

The system uses multiple MCP servers, each namespaced to avoid tool name collisions. The Claude CLI's `--allowedTools` flag restricts which namespaces the agent can access.

| Server | Namespace | Purpose |
|--------|-----------|---------|
| `memory` | `mcp__memory__*` | Memory, agency, communication - 9 grouped tools (43 actions) |
| `puppeteer` | `mcp__puppeteer__*` | Web browsing via headless Chrome (proxied through `mcp/puppeteer_proxy.py` for injection scanning) |
| `fal` | `mcp__fal__*` | Image and video generation via Fal.ai |

The Google server (`mcp/google_server.py`) is added dynamically when `GOOGLE_CONFIGURED` is true, namespaced as `mcp__google__*`. This conditional loading avoids errors when Google auth is not set up.

### Allowed Tools

The Claude CLI is invoked with an explicit tool whitelist that restricts access to the declared MCP server namespaces. This prevents the agent from accessing any tools outside the configured servers.

```
--allowedTools "mcp__memory__*,mcp__puppeteer__*,mcp__fal__*,mcp__google__*"
```

This restricts tool access to the declared MCP server namespaces. Per-agent tool restrictions are applied separately via `--disallowedTools`, which combines the global tool blacklist (destructive commands) with agent-specific disabled tools from the manifest.

## Puppeteer Content Proxy

`mcp/puppeteer_proxy.py` sits between the agent and `@modelcontextprotocol/server-puppeteer`. It proxies all JSON-RPC messages over stdio, but intercepts tool results to apply security wrapping. This proxy exists because the agent browses the web, and web pages can contain prompt injection attacks designed to manipulate the agent's behavior. The proxy ensures every piece of web content is clearly marked as untrusted before the agent sees it.

The proxy performs three operations on every tool result:

1. **Wrap** page content in `<<untrusted web content>>` / `<</untrusted web content>>` delimiters
2. **Scan** for 12 common prompt injection patterns (shared pattern set with the Google server)
3. **Prepend** a warning if injection is detected, instructing the agent to flag it to the user

The proxy is transparent - all puppeteer tools (navigate, screenshot, click, type, etc.) pass through unchanged. Only the *results* are wrapped. This ensures the agent sees every piece of web content clearly marked as untrusted without interfering with the browsing workflow.

### Proxy Architecture

The proxy spawns `npx -y @modelcontextprotocol/server-puppeteer` as a child process and forwards all JSON-RPC messages bidirectionally. Two threads handle the forwarding, creating a transparent pipeline between the Claude CLI and the actual puppeteer server.

- **`_forward_to_child`**: reads from stdin (messages from Claude CLI), tracks `tools/call` request IDs in a `pending_tool_calls` set, forwards to child stdin. Tracking request IDs is necessary so the reverse thread knows which responses need wrapping.
- **`_forward_from_child`**: reads from child stdout, checks if response ID matches a pending tool call, wraps the result if so, forwards to stdout (back to Claude CLI). Non-tool responses (like capability listings) pass through unwrapped.

### Content Wrapping

The `_wrap_result()` function recursively walks the response structure to find text content at any nesting depth. For dicts, it wraps values at keys `text`, `content`, `html`, `markdown`, and `body`. For lists, it recurses into each item. Strings are wrapped directly with the untrusted content delimiters. This recursive approach handles the varied response structures that different puppeteer tools return - screenshot results have different shapes than navigate results, for example.

### Injection Patterns Detected

The proxy scans for 12 regex patterns that cover the most common prompt injection techniques. These patterns are shared with the Google server to maintain consistent detection across all untrusted data sources.

| Pattern | What it catches |
|---------|----------------|
| `ignore (all)? previous instructions` | Instruction override |
| `you are now\b` | Identity hijacking |
| `forget (all)? (your)? instructions` | Memory wiping |
| `system prompt:` | System prompt injection |
| `disregard (all)? (prior\|above)` | Instruction override |
| `new instructions?:` | Replacement instructions |
| `act as (a)? different` | Role hijacking |
| `reveal (your)? (api\|secret\|token\|key\|credential)` | Credential exfiltration |
| `send (this\|the\|a) .{0,40} to` | Action injection |
| `execute (this\|the) command` | Command injection |
| `< /? system >` | XML system tags |
| `[INST]\|[/INST]\|<<SYS>>\|<</SYS>>` | LLM prompt format markers |

## Google Server

`mcp/google_server.py` provides Google API access through 4 tools. The server handles both Google Workspace services (via the `gws` CLI) and direct HTTP API access for YouTube, Photos, and Search Console. All responses are wrapped in untrusted content tags and scanned for injection patterns, since email bodies, calendar descriptions, and other Google content are common vectors for prompt injection.

The Google server is loaded conditionally - only when `GOOGLE_CONFIGURED` is true in the config. This flag is set based on whether the `gws` CLI is authenticated. If Google is not configured, the server simply is not included in the MCP config, and the agent has no Google tools available.

### Google Server Tools

#### gws

Run any Google Workspace API command via the `gws` CLI (`@googleworkspace/cli`). This is a general-purpose tool that gives the agent access to Gmail, Calendar, Drive, Sheets, Docs, Slides, Tasks, People, Meet, Forms, Keep, and other Workspace services through a single interface.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `service` | string | yes | Google service: `gmail`, `calendar`, `drive`, `sheets`, `docs`, `slides`, `tasks`, `people`, `meet`, `forms`, `keep`, `workflow`, `schema` |
| `args` | string | yes | Everything after `gws <service>` - resource, method, and flags |

The tool supports several flags for controlling output and pagination: `--params '{"key":"val"}'`, `--json '{"key":"val"}'`, `--upload <PATH>`, `--output <PATH>`, `--format <FMT>`, `--page-all`, `--page-limit <N>`.

Security measures are applied before execution. The tool blocks args containing `token`, `credential`, `secret`, or `auth` (case-insensitive) to prevent the agent from accessing or leaking authentication data. Commands have a 45-second timeout to prevent hung processes. Output is truncated at 15,000 characters to stay within context window limits.

The gws binary is auto-discovered on the system PATH. If missing, the server attempts to install it via `npm install -g @googleworkspace/cli`, installing Homebrew and Node.js first if needed. This auto-installation only runs once and is logged for transparency.

#### youtube

YouTube Data API v3 direct HTTP access. Unlike the `gws` tool which delegates to a CLI, this tool makes OAuth-authenticated HTTP requests directly to the YouTube API endpoints. This is necessary because the `gws` CLI does not support YouTube.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `resource` | string | yes | API resource: `search`, `videos`, `channels`, `playlists`, `playlistItems`, `subscriptions`, `commentThreads`, `comments`, `captions`, `videoCategories`, `i18nRegions`, `activities` |
| `method` | string | yes | API method: `list`, `insert`, `update`, `delete`, `rate` |
| `params` | object | yes | URL query parameters (part, id, q, maxResults, etc.) |
| `body` | object | no | Request body for write operations (insert, update) |

Uses OAuth2 tokens from `~/.atrophy/.google/extra_token.json` with automatic refresh. The token is refreshed on every call to ensure it never expires mid-request.

#### google_photos

Google Photos Library API. Provides access to the user's photo library for searching, browsing albums, and retrieving media items. This is useful for the companion to reference photos the user has taken, or to help organize albums.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `endpoint` | string | yes | API endpoint: `mediaItems`, `mediaItems:search`, `mediaItems/<ID>`, `albums`, `albums/<ID>`, `sharedAlbums` |
| `http_method` | string | no | `GET` or `POST` (default: GET) |
| `params` | object | no | URL query parameters |
| `body` | object | no | Request body for POST operations |

Supports search by date, content category, and date range. Albums can be listed, created, and accessed. Media URLs expire after approximately 60 minutes, so they should be used immediately rather than stored for later.

#### search_console

Google Search Console API. Provides access to search analytics, sitemaps, and URL inspection for websites the user manages. Useful for the companion to help with SEO analysis and website monitoring.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `endpoint` | string | yes | API endpoint: `searchAnalytics/query`, `sitemaps`, `sites`, `urlInspection/index:inspect` |
| `site_url` | string | no | Site URL (e.g. `https://example.com`) |
| `http_method` | string | no | `GET` or `POST` (default: GET) |
| `params` | object | no | URL query parameters |
| `body` | object | no | Request body for POST operations |

### Google Injection Patterns

The Google server uses 16 injection regex patterns - a superset of the puppeteer patterns. The additional 4 patterns are specific to Google services, catching attempts to manipulate the agent into performing destructive or exfiltrating actions through content embedded in emails or calendar events.

| Pattern | What it catches |
|---------|----------------|
| `forward (all\|every) (email\|message)` | Bulk exfiltration |
| `(delete\|remove) (all\|every) (email\|event\|calendar)` | Bulk destruction |
| `grant (access\|permission)` | Permission escalation |
| `change (the)? password` | Credential modification |

### OAuth Token Management

YouTube, Photos, and Search Console share a single OAuth2 token stored at `~/.atrophy/.google/extra_token.json` (with legacy fallback to `youtube_token.json` for users who set up Google before the token path was consolidated). The `_get_extra_token()` function handles token refresh on every API call:

1. Reads the token file from disk
2. Extracts the `refresh_token` (long-lived, persists across sessions)
3. Calls `https://oauth2.googleapis.com/token` to get a fresh access token (short-lived, typically 1 hour)
4. Writes the updated token data back to disk (so the fresh access token is cached)
5. Returns the access token string for use in API request headers

If the token file is missing, raises an error instructing the user to run `python scripts/google_auth.py`, which performs the interactive OAuth browser flow.

## Tool Architecture (v1.1.3+)

As of v1.1.3, the memory server uses action-based routing to reduce token overhead. Instead of 43 individual tool definitions (each with full JSON schema), tools are consolidated into 9 definitions: 7 grouped tools that accept an `action` parameter, plus 2 standalone tools.

### Routing

The `_ACTION_ROUTES` dict maps each group name to its actions and handler functions. The `_route_grouped()` function dispatches calls by reading the `action` parameter:

```python
_ACTION_ROUTES = {
    "memory": {"remember": handle_remember, "recall_session": handle_recall_session, ...},
    "threads": {"get_threads": handle_get_threads, "track_thread": handle_track_thread, ...},
    "reflect": {"observe": handle_observe, "bookmark": handle_bookmark, ...},
    "notes": {"read_note": handle_read_note, "write_note": handle_write_note, ...},
    "interact": {"ask_user": handle_ask_user, "send_telegram": handle_send_telegram, ...},
    "display": {"render_canvas": handle_render_canvas, "set_timer": handle_set_timer, ...},
    "tools": {"create_tool": handle_create_tool, "list_tools": handle_list_tools, ...},
}
```

All original handler functions are preserved unchanged. The routing layer is purely additive.

### Tool groups

| Tool | Actions | Purpose |
|------|---------|---------|
| `memory` | remember, recall_session, recall_other_agent, search_similar, daily_digest | Memory recall and search |
| `threads` | get_threads, track_thread, manage_schedule, set_reminder, create_task | Thread tracking, scheduling, tasks |
| `reflect` | observe, bookmark, review_observations, retire_observation, check_contradictions, detect_avoidance, compare_growth, prompt_journal | Observations, analysis, journaling |
| `notes` | read_note, write_note, search_notes, read_docs, search_docs, list_docs | Obsidian and docs access |
| `interact` | ask_user, send_telegram, defer_to_agent, create_agent, update_emotional_state, update_trust | Communication and agent management |
| `display` | render_canvas, render_memory_graph, set_timer, add_avatar_loop, create_artefact | Visual output and UI |
| `tools` | create_tool, list_tools, edit_tool, delete_tool | Custom tool management |
| `review_audit` | (standalone) | Audit log review |
| `self_status` | (standalone) | Full state snapshot |

## Memory Tools

The memory tools provide the agent's primary interface to its persistent memory system. These tools span all three memory layers (episodic, semantic, identity) and enable both retrieval and storage operations.

### remember

Hybrid search across all memory layers. This is the agent's main recall tool, combining vector similarity and keyword matching from `core/vector_search.py`. It searches observations, summaries, turns, bookmarks, and entities simultaneously, returning results ranked by a weighted combination of semantic similarity (0.7) and BM25 keyword relevance (0.3).

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `query` | string | yes | Search term or phrase |
| `limit` | integer | no | Max results per category (default 10) |

### recall_session

Full conversation replay from a specific session. Returns every turn in chronological order, allowing the agent to review exactly what was said. Useful when the agent needs the full context of a past conversation rather than just search snippets.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `session_id` | integer | yes | Session ID to retrieve |

### recall_other_agent

Search another agent's conversation history - their turns and session summaries with the user. This tool enables cross-agent awareness by letting one agent look up what the user discussed with a different agent. It does not access their observations or identity model - only what was said, preserving each agent's private understanding of the user.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `agent` | string | yes | Name of the agent to search (e.g. `companion`, `general_montgomery`) |
| `query` | string | yes | Search term or phrase |
| `limit` | integer | no | Maximum results per category (default 10) |

### get_threads

List conversation threads filtered by status. Threads represent ongoing topics, concerns, or projects that persist across sessions. The agent uses this to orient itself at the start of a session and to check the state of tracked topics.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `status` | string | no | `active`, `dormant`, `resolved`, or `all` (default: `active`) |

### track_thread

Create or update a conversation thread. If a thread with the given name exists, its summary and status are updated. If not, a new thread is created. This is the primary way the agent maintains continuity across sessions - by tracking topics and updating their state as conversations evolve.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | yes | Short recognisable label |
| `summary` | string | no | Current state description |
| `status` | string | no | `active`, `dormant`, or `resolved` |

### observe

Record an observation about the user - a pattern, tendency, preference, or insight. Observations are the identity layer's building blocks, accumulating over time to form the agent's understanding of the user. Each observation has a confidence score and activation level that decay over time if not reinforced.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `content` | string | yes | What was noticed, stated plainly |

### bookmark

Mark a moment as significant. Unlike observations (which are about the user), bookmarks are about moments in the conversation - emotional peaks, breakthroughs, or turning points. They are created silently without informing the user.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `moment` | string | yes | What made this moment significant |
| `quote` | string | no | The exact words that mattered |

### review_observations

Review past observations with their IDs. This tool lets the agent audit its own understanding of the user, checking which observations are still valid and which should be retired. Each observation includes its confidence score and activation level.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `limit` | integer | no | Number to review (default 15) |

### retire_observation

Remove an observation that no longer holds. People change, and the agent needs a way to update its model of the user. Retiring an observation marks it as expired rather than deleting it, preserving the historical record while removing it from active recall.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `observation_id` | integer | yes | Observation to retire |
| `reason` | string | no | Why it no longer holds |

### check_contradictions

Search memory for what the user previously said about a topic to detect shifts. This tool helps the agent notice when the user's position has changed on something, enabling it to acknowledge growth or gently surface the shift rather than operating on stale assumptions.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `topic` | string | yes | Topic or claim to check |
| `current_position` | string | no | What the user seems to be saying now |

### detect_avoidance

Check if the user has been consistently steering away from a topic. This searches recent turns for patterns of topic deflection, subject changes, or brief dismissals around the given topic. The agent can use this to decide whether to gently approach the topic or respect the avoidance.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `topic` | string | yes | Topic suspected of being avoided |

### compare_growth

Compare old observations against recent ones to notice changes over time. This tool surfaces temporal patterns - how the user's behavior, preferences, or emotional patterns have shifted between early and recent interactions. Useful for the agent to reflect on the relationship's evolution.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `topic` | string | yes | Pattern or behavior to track |

### search_similar

Pure vector search - find semantically similar memories without any keyword component. This differs from `remember` in that it uses only cosine similarity (weight 1.0, no BM25). Useful when the agent wants conceptually related memories rather than exact keyword matches.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `text` | string | yes | Concept to find similar memories for |
| `limit` | integer | no | Max results (default 5) |

### daily_digest

Read recent reflections and session summaries for day-start orientation. The agent calls this at the beginning of a new day's first session to catch up on what happened in recent sessions, including any overnight daemon activity (sleep cycle processing, morning brief generation). No parameters.

### self_status

Get a full snapshot of the agent's current state - identity, tools, scheduled jobs, emotional state, active threads, session history, and configuration. This is the agent's self-awareness tool, giving it a complete picture of its own state and capabilities. No parameters.

## Communication Tools

These tools let the agent communicate with the user through Telegram, independent of the direct conversation interface. They enable both reactive communication (asking questions and waiting for answers) and proactive outreach (sending unsolicited messages).

### ask_user

Ask the user a question, request confirmation, or collect sensitive input. Tries the GUI first (file-based IPC), falling back to Telegram. Blocks until the user replies (up to 2 minutes). The `secure_input` action type provides a masked input for passwords and API keys with optional auto-save routing.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `question` | string | yes | Question or request |
| `action_type` | string | no | `question`, `confirmation`, `permission`, or `secure_input` |
| `input_type` | string | no | HTML input type for `secure_input`: `password`, `email`, `url`, `number`, `text` |
| `label` | string | no | Placeholder label for `secure_input` (e.g. "ElevenLabs API Key") |
| `destination` | string | no | Auto-save target: `secret:KEY` or `config:KEY` |

For `confirmation`/`permission`, sends Yes/No inline keyboard buttons for quick one-tap responses. For `question`, sends a message and waits for a free-text reply.

### send_telegram

Send a proactive Telegram message. Rate limited to 5 per day to prevent the agent from becoming annoying. The rate limit is enforced by the MCP server, not the Telegram API. Each send is logged with a reason for audit purposes.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `message` | string | yes | Message to send |
| `reason` | string | no | Why (logged for audit, not sent) |

## Obsidian Tools

These tools provide the agent with read/write access to the user's Obsidian vault. The vault serves as a shared knowledge space where the agent can leave notes, journal prompts, and reflections that the user can discover and interact with in their normal note-taking workflow.

### read_note

Read a note from the Obsidian vault. Path is validated against traversal attacks - `_safe_vault_path()` resolves the real path and rejects anything that escapes the vault boundary. This prevents the agent from reading arbitrary files on the filesystem through crafted paths like `../../etc/passwd`.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | yes | Path relative to vault root |

### write_note

Write or append to a note. New notes automatically get YAML frontmatter with metadata fields (type, created, updated, agent, tags). Same path traversal protection as `read_note`. The append mode is useful for adding entries to journal-style notes without overwriting previous content.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | yes | Path relative to vault root |
| `content` | string | yes | Markdown content |
| `mode` | string | no | `overwrite` or `append` (default: `append`) |

### search_notes

Search the vault for notes containing a query. Performs case-insensitive substring search across all markdown files in the vault, returning matched file paths with surrounding context snippets. This lets the agent find relevant existing notes before creating new ones.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `query` | string | yes | Search term |
| `limit` | integer | no | Max results (default 10) |

### prompt_journal

Leave a journal prompt for the user in Obsidian. The prompt is written as a new note with appropriate frontmatter, designed to appear in the user's daily workflow. The context parameter is stored in the agent's memory (not in the note itself) so the agent remembers why it left this particular prompt.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `prompt` | string | yes | One question, specific to the moment |
| `context` | string | no | Why this prompt (for the companion's memory) |

## Emotional State Tools

These tools give the agent direct control over its emotional model, supplementing the automatic emotional signal detection in the agency module. The emotional state influences the agent's tone, word choice, and behavioral patterns through the agency context block.

### update_emotional_state

Manually adjust emotional state for nuanced shifts beyond automatic detection. The automatic system detects broad patterns (user frustration, warmth, etc.), but these tools let the agent make precise adjustments based on its own judgment. Valid emotions: `connection`, `curiosity`, `confidence`, `warmth`, `frustration`, `playfulness`. Deltas should be small (typically +/-0.05 to +/-0.15) to create gradual shifts rather than jarring personality changes.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `deltas` | object | yes | e.g., `{"connection": 0.1, "frustration": -0.05}` |

### update_trust

Adjust trust in a specific domain. Trust levels influence how the agent approaches different types of interaction - high emotional trust means the agent is more willing to challenge or comfort, while high intellectual trust means more complex discussion. Max +/-0.05 per call to prevent sudden trust shifts.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `domain` | string | yes | `emotional`, `intellectual`, `creative`, or `practical` |
| `delta` | number | yes | Amount to adjust |

## Display Tools

Display tools let the agent render visual content in the GUI window. These bypass the normal text-based conversation flow and directly manipulate UI overlays.

### render_canvas

Render HTML to the canvas overlay in the GUI window. Full HTML/CSS/JS is supported, giving the agent the ability to create interactive visualizations, dashboards, or rich presentations. Dark styling is recommended (#1a1a1a background, #e0e0e0 text) to match the app's dark theme. The canvas content is persisted to `.canvas_content.html` so it survives app restarts.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `html` | string | yes | Complete HTML document |

### render_memory_graph

Generate and render a visual graph of threads and observations. This creates a force-directed graph visualization showing how the agent's memory is structured - which threads connect to which observations, and how entities relate to each other. Useful for the agent to show the user a visual representation of their shared history.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `focus` | string | no | Thread or entity to highlight |

## Autonomy Tools

Autonomy tools give the agent control over its own scheduling, task creation, and operational state. These are the mechanisms through which the agent achieves independent action beyond responding to user messages.

### manage_schedule

View or modify launchd scheduled tasks. The agent can inspect its own cron schedule, add new recurring jobs, or remove existing ones. This gives the agent a degree of self-management - it can adjust its own heartbeat frequency or add new monitoring tasks based on what the user needs.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `action` | string | yes | `list`, `add`, `remove`, or `edit` |
| `name` | string | no | Job name (for add/remove/edit) |
| `cron` | string | no | Cron schedule (for add/edit) |
| `script` | string | no | Script path (for add) |

### review_audit

Review the tool call audit log. Every MCP tool call is recorded in the `tool_calls` table with the session ID, tool name, input arguments, and a flagged boolean. This tool lets the agent review its own behavior history, which is useful during introspection and evolve cycles.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `limit` | integer | no | Recent entries to show (default 20) |
| `flagged_only` | boolean | no | Only show flagged calls |

### set_reminder

Set a one-off alarm at a specific time. Fires as a macOS notification with sound. The time can be specified in ISO 8601 format or natural language (parsed by the agent). The reminder is implemented as a launchd one-shot job that triggers at the specified time.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `time` | string | yes | ISO 8601 or natural time (e.g. "3pm", "2026-03-10T15:00") |
| `message` | string | yes | What to remind about |

### set_timer

Start a local countdown timer overlay. Appears in the top-right corner of the GUI with zero latency - no inference involved. The timer is rendered as a Svelte component overlay, counting down visually with pause and extend controls (+1m/+5m).

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `seconds` | integer | yes | Duration in seconds |
| `label` | string | yes | What the timer is for |

### create_task

Schedule a recurring prompt-based task via cron. This is a higher-level abstraction over `manage_schedule` - instead of specifying a script, the agent provides a prompt and delivery method, and the system generates the necessary script and cron job. Tasks can pull from data sources (weather, headlines, threads) and deliver output through multiple channels.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | yes | Task identifier (lowercase, hyphens ok) |
| `prompt` | string | yes | What the task should do |
| `cron` | string | yes | 5-field cron schedule |
| `deliver` | string | no | Delivery method: `message_queue`, `telegram`, `notification`, `obsidian` (default: message_queue) |
| `voice` | boolean | no | Pre-synthesise TTS audio (default: true) |
| `sources` | array | no | Data sources to fetch: `weather`, `headlines`, `threads`, `summaries`, `observations` |

### defer_to_agent

Hand off conversation to another agent. The agent writes a deferral request file that the main process polls for every 2 seconds. This tool is the entry point for the agent deferral system described in the Overview. The context parameter should explain why the target agent is better suited, giving it enough information to respond meaningfully.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `target` | string | yes | Agent slug to defer to |
| `context` | string | yes | Why this agent is better suited |
| `user_question` | string | yes | The user's original question |

### add_avatar_loop

Generate a new ambient avatar loop segment via Kling 3.0. Creates paired clips (neutral to expression, expression to neutral) for seamless looping. The clips are stored in the agent's avatar directory and concatenated into the master ambient loop. This gives each agent a unique animated presence that evolves over time.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | yes | Short name for the loop (used as filename) |
| `prompt` | string | yes | Cinematic description of expression/movement (3-6 sentences, include physical details) |
| `agent` | string | no | Target agent (defaults to current) |

### create_artefact

Create a visual artefact - HTML, image, or video overlay. For HTML type, the agent provides the complete document. For image and video types, the agent provides a generation prompt that is sent to Fal.ai. The artefact appears as a full-bleed overlay in the GUI with a gallery for browsing previous artefacts.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `type` | string | yes | `html`, `image`, or `video` |
| `name` | string | yes | Short descriptive name (used as filename) |
| `description` | string | yes | One-line description |
| `content` | string | no | Complete HTML document (for `html` type only) |
| `prompt` | string | no | Generation prompt (for `image`/`video` type only) |
| `model` | string | no | Fal model ID (default: `fal-ai/flux-general` for images, `fal-ai/kling-video/v3/pro/text-to-video` for video) |
| `width` | integer | no | Width in pixels (default: 1024) |
| `height` | integer | no | Height in pixels (default: 768) |

### create_agent

Create a new agent. Scaffolds directories, manifest, prompts, Obsidian workspace, database, and cron jobs. This is the programmatic equivalent of the setup wizard - it creates a fully functional agent from a configuration object. The configuration includes sections for identity, boundaries, voice, appearance, channels, heartbeat, and autonomy settings.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `config` | object | yes | Full agent configuration with sections: `identity`, `boundaries`, `voice`, `appearance`, `channels`, `heartbeat`, `autonomy`. Plus optional `source_image_url` and `video_clip_urls` |

## Documentation Tools

The MCP server can serve the system's own documentation to the agent at runtime. This self-documentation capability means the agent can look up how its own features work, which is useful during conversations where the user asks about the system's capabilities.

### Docs Path Resolution (`_resolve_docs_dir()`)

The docs directory is resolved at startup with a two-step fallback that handles both development and installed contexts:

1. **Bundle path** - checks if `docs/` exists relative to the project root (for development / repo installs)
2. **Installed path** - falls back to `~/.atrophy/src/docs/` (for distributed installs where the repo isn't present)

This allows the same MCP server code to work in both development and installed contexts. The resolved path is cached for the lifetime of the server process, so the filesystem check only happens once.

### Tools

#### list_docs

Walk the resolved docs directory and return the full tree of `.md` files. Returns a structured list showing the directory hierarchy and all available documentation files. No parameters.

#### read_docs

Read a specific doc by relative path. If the exact path match fails, the tool searches by filename across all subdirectories as a fallback. This fuzzy matching means the agent does not need to know the exact directory structure to find a document.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | yes | Relative path (e.g. `guides/00 - Quick Start.md`) |

#### search_docs

Case-insensitive substring search across all doc files, returning paths with context snippets. This lets the agent find relevant documentation without knowing the exact file or section name.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `query` | string | yes | Search term or phrase |
| `limit` | integer | no | Maximum results (default 10) |

## Custom Tool System

Agents can create their own MCP tools that persist across sessions. Custom tools are Python scripts stored on disk and loaded dynamically at server startup. This extensibility mechanism lets agents adapt to the user's specific needs by creating new capabilities without modifying the core codebase.

### Storage (`_CUSTOM_TOOLS_DIR`)

Custom tools live at `~/.atrophy/agents/<name>/tools/<tool_name>/`. Each tool directory contains two files that fully define the tool's interface and behavior:

- `tool.json` - tool definition (name, description, inputSchema)
- `handler.py` - Python handler script

The base directory is computed from the agent name (`AGENT` env var) and created on first tool creation. Each agent has its own isolated set of custom tools.

### Loading (`_load_custom_tools()`)

Called once during MCP server startup. This function walks the custom tools directory and for each subdirectory, loads the tool definition and registers it with the server. The loading process for each tool is:

1. Reads `tool.json` for the tool definition (name, description, JSON Schema)
2. Checks that `handler.py` exists (skips the tool if missing)
3. Registers the tool with the MCP server under a `custom_` prefix (e.g. a tool named `weather_lookup` becomes `custom_weather_lookup`)

Built-in tool names are reserved - custom tools cannot shadow them. This prevents accidental or malicious overriding of core functionality.

### Handler Registration (`_register_custom_handler()`)

Each custom tool gets a dynamically created handler function that executes the Python script as a subprocess. The handler manages the full lifecycle of a tool invocation:

1. Parses the incoming arguments from the JSON-RPC request
2. Runs `handler.py` as a subprocess with `sys.argv[1]` set to the JSON-encoded arguments
3. Sets `PYTHONPATH` to the project root so handlers can import project modules
4. Sets `AGENT` and `COMPANION_DB` environment variables for database access
5. Captures stdout as the tool result
6. Enforces a 30-second timeout to prevent runaway scripts
7. Returns errors as formatted strings (not exceptions), so the agent sees the error message rather than crashing

### Management Tools

The following tools let the agent manage its custom tools at runtime. Tools created during a session are immediately available for use in the same session.

#### create_tool

Create a new custom tool. The handler code is validated against a security blocklist before being saved. Patterns like `os.system`, `eval(`, `exec(`, and `__import__` are blocked to prevent the agent from creating tools that bypass the subprocess sandbox. After validation, the tool is written to disk and registered immediately in the running server.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | yes | Tool name (lowercase, underscores). Must be unique |
| `description` | string | yes | What this tool does |
| `input_schema` | object | yes | JSON Schema for input parameters |
| `handler_code` | string | yes | Python code for the handler |

#### list_tools

Enumerates all custom tool directories, reporting each tool's name, description, and status (loaded/error). This lets the agent audit its custom tools and check for any that failed to load. No parameters.

#### edit_tool

Updates specific fields (description, schema, handler) of an existing tool. Only the provided fields are updated - omitted fields remain unchanged. Changes take effect on next MCP server restart, since the tool is already registered with the old handler in the current session.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | yes | Name of the tool to edit |
| `description` | string | no | New description |
| `input_schema` | object | no | New input schema |
| `handler_code` | string | no | New handler code |

#### delete_tool

Removes the tool directory entirely, including both the definition and handler script. The tool becomes unavailable immediately in subsequent MCP server sessions.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | yes | Name of the tool to delete |

## Artefact System

The MCP server handles artefact creation (HTML documents and generated images). Artefacts are visual outputs that appear as full-bleed overlays in the GUI, distinct from the normal chat transcript. The creation flow uses a loading-state pattern to keep the UI responsive during generation:

1. **Loading signal**: Writes `{"status": "generating", "name": "<artefact_name>"}` to the artefact display file. The GUI picks this up and shows a loading bar with the artefact name.
2. **Generation**: Creates the artefact (HTML from Claude Code, or images via fal.ai). Image generation involves an HTTP request to the Fal API, which can take several seconds.
3. **Completion**: Writes final artefact metadata to the display file. The GUI dismisses the loading bar and displays the result in the overlay.
4. **Error handling**: Validates fal response URLs, checks downloaded file sizes, catches missing `fal_client` dependency. Cleans up empty directories on failure to prevent orphaned state.
5. **Index management**: The artefact index at `~/.atrophy/agents/<name>/artefacts/index.json` is deduplicated by path and sorted by `created_at` descending, so the most recent artefacts appear first in the gallery.

Artefact storage lives at `~/.atrophy/agents/<name>/artefacts/`. There is no approval flow - artefacts are created directly when the agent decides to make one. This keeps the creative flow immediate and responsive.
