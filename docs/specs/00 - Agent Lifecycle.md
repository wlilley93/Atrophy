# Agent Lifecycle

This specification describes the end-to-end lifecycle of a companion agent session, from startup through shutdown and inter-session autonomous activity. It covers the agent manifest schema, discovery algorithm, deferral system, emotional state model, and session management. Each phase is documented with its implementation details, design rationale, and connections to adjacent systems.

---

## 1. Startup

The application startup sequence initializes configuration, databases, IPC channels, background services, and the UI window. The entry point is `src/main/index.ts`, which supports three operating modes selected via command-line arguments: `--app` (menu bar), GUI (default, Svelte window), and `--server` (HTTP API, no window). The mode determines which UI elements are created but does not affect the core initialization sequence.

### Sequence

1. **Electron ready**: `app.whenReady()` fires, signalling that Electron's internal initialization is complete and the app can begin creating windows and registering handlers. Command-line arguments are parsed to determine the operating mode. In menu bar or server mode, `app.dock.hide()` is called to remove the dock icon since these modes operate as background processes. In GUI mode, the dock icon is set to a brain `.icns` from `resources/icons/TheAtrophiedMind.icns`, with a procedural orb fallback generated by `getAppIcon()` if the icon file is missing.

2. **Load configuration**: `ensureUserData()` creates `~/.atrophy/` on first run with subdirectories for agents, logs, and models. It writes an empty `config.json` (mode 0600) if none exists. Then `migrateAgentData()` copies bundled agent files to the user data directory, skipping files that already exist and always skipping `agent.json` manifests (so user customizations are never overwritten). `getConfig()` instantiates the Config singleton, which performs the following steps:
   - Loads `~/.atrophy/.env` into `process.env` (simple key=value parsing, quotes stripped, existing env vars not overwritten)
   - Reads `~/.atrophy/config.json` for user settings
   - Resolves the version from `BUNDLE_ROOT/VERSION`
   - Resolves the agent (default: `xan`) via `_resolveAgent()` which loads the agent manifest and computes all derived paths (database path, prompt directories, emotional state file, etc.)
   - Detects the Python path by checking `PYTHON_PATH` env var, then `python3`, `/opt/homebrew/bin/python3`, `/usr/local/bin/python3` in order

3. **Initialize database**: `initDb()` executes `db/schema.sql` to create tables (idempotent via `IF NOT EXISTS`), then runs migrations for schema evolution on existing databases. The database uses `better-sqlite3` with its synchronous API, WAL journal mode, and foreign keys enabled.

4. **Register IPC handlers**: `registerIpcHandlers()` sets up all `ipcMain.handle()` channels that bridge the main and renderer processes. These channels are the only way the renderer can interact with the main process, enforcing the security boundary defined by `contextBridge`. The full channel list is organized by namespace:

   - Config: `config:get`, `config:update`
   - Agents: `agent:list`, `agent:listFull`, `agent:cycle`, `agent:switch`, `agent:getState`, `agent:setState`
   - Inference: `inference:send`, `inference:stop`
   - Setup: `setup:check`, `setup:inference`, `setup:saveSecret`, `setup:createAgent`
   - Window: `window:toggleFullscreen`, `window:minimize`, `window:close`
   - Usage: `usage:all`, `activity:all`
   - Opening: `opening:get`
   - Avatar: `avatar:getVideoPath`
   - Cron: `cron:list`, `cron:toggle`
   - Telegram: `telegram:startDaemon`, `telegram:stopDaemon`
   - Server: `server:start`, `server:stop`
   - Memory: `memory:search`
   - Login: `install:isEnabled`, `install:toggle`
   - Updater: `updater:check`, `updater:download`, `updater:quitAndInstall`
   - Deferral: `deferral:complete`
   - Queue: `queue:drainAgent`, `queue:drainAll`

   Audio-specific channels (`audio:start`, `audio:stop`, `audio:chunk`, etc.) are registered separately by `registerAudioHandlers()` and `registerWakeWordHandlers()`.

5. **Configure TTS callbacks**: `setPlaybackCallbacks()` wires TTS playback events to IPC sends, connecting the audio pipeline to the renderer's playback state. Three callbacks are registered:
   - `onStarted(index)` - sends `tts:started` to renderer, pauses wake word listener so the agent's own speech does not trigger wake detection
   - `onDone(index)` - sends `tts:done` to renderer so it can update playback indicators
   - `onQueueEmpty()` - sends `tts:queueEmpty` to renderer, resumes wake word listener since the agent has finished speaking

6. **Resume last agent**: `getLastActiveAgent()` reads `_last_active` from `~/.atrophy/agent_states.json` to determine which agent was active when the app last quit. If a different agent was last active (not the default `xan`), `config.reloadForAgent()` and `initDb()` switch to it. This ensures the app resumes where the user left off rather than always starting with the default agent.

7. **System prompt**: The system prompt is not loaded during startup. It is loaded lazily on the first `inference:send` call via `loadSystemPrompt()` in `src/main/context.ts`. This deferred loading avoids unnecessary file I/O on startup and ensures the prompt reflects the current agent's configuration. The resolution order is:
   1. `loadPrompt('system', '')` - searches tiered directories for `system.md` (Obsidian vault, user agent dir, bundle agent dir)
   2. `agents/<name>/prompts/system_prompt.md` - bundle fallback
   3. `'You are a companion. Be genuine, direct, and honest.'` - hardcoded fallback

   After loading the base prompt, skill files are appended (soul.md, heartbeat.md, gift.md, etc.) to give the agent its full behavioral repertoire, and the agent roster is appended for deferral awareness (see Section 11).

8. **Start background timers**: Three interval timers run throughout the application's lifetime to maintain conversational quality, process queued messages, and handle agent handoffs:
   - **Sentinel**: Coherence check every 5 minutes (`setInterval(300_000)`) - monitors the companion's recent output for repetition, flatness, and drift
   - **Queue poller**: Drains `.message_queue.json` every 10 seconds (`setInterval(10_000)`) - picks up messages from background daemons and forwards them to the renderer
   - **Deferral watcher**: Checks for agent handoff requests every 2 seconds (`setInterval(2_000)`) - reads `.deferral_request.json` and initiates agent switches

9. **Create window**: The BrowserWindow is created with settings tuned for the app's transparent, vibrancy-based design. The window uses macOS native vibrancy rather than a custom dark background, giving it the system's blur-behind-window effect.

   ```
   width: config.WINDOW_WIDTH (default 622)
   height: config.WINDOW_HEIGHT (default 830)
   minWidth: 360, minHeight: 480
   titleBarStyle: 'hiddenInset'
   trafficLightPosition: { x: 14, y: 14 }
   vibrancy: 'ultra-dark'
   visualEffectState: 'active'
   backgroundColor: '#00000000' (transparent)
   contextIsolation: true
   nodeIntegration: false
   webSecurity: false
   sandbox: false
   ```

   In menu bar mode, the window starts hidden and a `Tray` is created with a `Cmd+Shift+Space` global shortcut for toggling visibility. This lets the app operate as a persistent background companion accessible via keyboard shortcut.

10. **Initialize auto-updater**: `initAutoUpdater()` sets up `electron-updater` to check for new releases on GitHub. This step is skipped in dev mode (detected via `ELECTRON_RENDERER_URL`). The updater checks for updates after a 5-second delay to avoid blocking startup, and all update actions require explicit user consent.

### First Inference

When the renderer mounts, it calls `opening:get` via IPC, which returns the static opening line from the agent manifest (or the fallback `'Ready. Where are we?'`). This opening line is displayed immediately without inference. The `Session` object is not created until the first user message arrives via `inference:send`, which means no database writes, CLI spawns, or MCP connections happen until the user actively engages.

## 2. Agent Manifest Schema

The agent manifest is a JSON file that defines an agent's identity, voice, behavior, and service configuration. It lives at `agents/<name>/data/agent.json` in both the bundle and the user data directory. The user version takes precedence for user-modified fields (like voice settings changed through the Settings panel), but the bundle version is the source of truth for the base manifest.

### Full schema

The following table documents every field in the manifest. The "Resolution" column indicates which config function resolves the field - `cfg` for user-first resolution (env > config > manifest > default) and `agentCfg` for agent-first resolution (manifest > env > config > default). Fields marked "manifest only" are read directly from the JSON without the config resolution chain.

| Field | Type | Default | Resolution | Description |
|-------|------|---------|------------|-------------|
| `display_name` | `string` | Capitalized `name` | `agentCfg` | Human-readable name shown in UI |
| `user_name` | `string` | `"User"` | `cfg` | Name of the user |
| `OPENING_LINE` | `string` | `"Hello."` | `agentCfg` | First words the agent says |
| `wake_words` | `string[]` | `["hey <name>", "<name>"]` | manifest only | Trigger phrases for wake word detection |
| `telegram_emoji` | `string` | `""` | manifest only | Emoji prefix for Telegram messages |
| `disabled_tools` | `string[]` | `[]` | manifest only | MCP tools to disable for this agent |
| `description` | `string` | `""` | manifest only | Brief description (used in roster/deferral context) |
| `role` | `string` | `""` | manifest only | `"system"` for system agents (sorted first in discovery) |
| `TTS_BACKEND` | `string` | `"elevenlabs"` | `agentCfg` | TTS engine: `"elevenlabs"`, `"fal"`, `"say"`, `"off"` |
| `ELEVENLABS_VOICE_ID` | `string` | `""` | `agentCfg` | ElevenLabs voice ID |
| `ELEVENLABS_MODEL` | `string` | `"eleven_v3"` | `agentCfg` | ElevenLabs model |
| `ELEVENLABS_STABILITY` | `number` | `0.5` | `agentCfg` | Voice stability (0.0-1.0) |
| `ELEVENLABS_SIMILARITY` | `number` | `0.75` | `agentCfg` | Voice similarity boost (0.0-1.0) |
| `ELEVENLABS_STYLE` | `number` | `0.35` | `agentCfg` | Voice style exaggeration (0.0-1.0) |
| `TTS_PLAYBACK_RATE` | `number` | `1.12` | `agentCfg` | TTS playback speed |
| `FAL_VOICE_ID` | `string` | `""` | `agentCfg` | Fal.ai voice ID (fallback TTS) |
| `HEARTBEAT_ACTIVE_START` | `number` | `9` | `agentCfg` | Heartbeat active hours start (24h) |
| `HEARTBEAT_ACTIVE_END` | `number` | `22` | `agentCfg` | Heartbeat active hours end (24h) |
| `HEARTBEAT_INTERVAL_MINS` | `number` | `30` | `agentCfg` | Minutes between heartbeat checks |
| `TELEGRAM_BOT_TOKEN` | `string` | `""` | `agentCfg` | Per-agent Telegram bot token |
| `TELEGRAM_CHAT_ID` | `string` | `""` | `agentCfg` | Per-agent Telegram chat ID |
| `WINDOW_WIDTH` | `number` | `622` | `agentCfg` | Window width in pixels |
| `WINDOW_HEIGHT` | `number` | `830` | `agentCfg` | Window height in pixels |
| `DISABLED_TOOLS` | `string[]` | `[]` | `agentCfg` | Tools to block (saved via settings) |

### Resolution modes

The two resolution functions serve different use cases. `cfg` gives user-level settings priority, which is appropriate for settings the user controls globally (like their name). `agentCfg` gives the agent manifest priority, which is appropriate for per-agent identity settings (like voice and display name) that should differ between agents even when the same user operates all of them.

- **`cfg(key, default)`**: env var > user config > agent manifest > default (user-level settings)
- **`agentCfg(key, default)`**: agent manifest > env var > user config > default (agent-level settings take priority)

## 3. Agent Discovery Algorithm

Agent discovery builds the list of available agents by scanning two directories and merging the results. The implementation lives in `discoverAgents()` in `src/main/agent-manager.ts`. This function is called when listing agents in the UI, cycling between agents, and building the roster for deferral awareness.

### Search directories

Two directories are searched in order, with the user directory taking precedence over the bundle directory. This allows users to install custom agents alongside the bundled defaults.

1. `~/.atrophy/agents/` (user-installed agents)
2. `<BUNDLE_ROOT>/agents/` (bundled agents)

The bundle directory is skipped if it resolves to the same path as the user directory, which happens during development when `BUNDLE_ROOT` points to the project root.

### Discovery steps

The algorithm walks each search directory and builds the agent list with deduplication. User-directory agents shadow bundle agents with the same name, allowing users to override bundled agent configurations.

1. Read directory entries from each search directory, sorted alphabetically
2. For each entry, check if `<entry>/data/` exists (not `agent.json` specifically - just the `data` directory). This is a lightweight check that avoids reading JSON files for entries that are not agent directories.
3. Skip entries already seen (user directory takes precedence over bundle)
4. Load the manifest from `<entry>/data/agent.json` to extract `display_name`, `description`, and `role`
5. If no manifest is found, use defaults (capitalized name, empty description, empty role)

### Sort order

Agents are sorted with system-role agents first, then alphabetically by name. System agents (like `xan`) appear at the top of the list because they serve as the default entry point and should be easily accessible.

```typescript
agents.sort((a, b) => {
  const aSystem = a.role === 'system' ? 0 : 1;
  const bSystem = b.role === 'system' ? 0 : 1;
  if (aSystem !== bSystem) return aSystem - bSystem;
  return a.name.localeCompare(b.name);
});
```

### Agent cycling

`cycleAgent(direction, current)` walks the sorted agent list in the given direction (+1 or -1), skipping disabled agents. It wraps around the list using modular arithmetic so cycling past the last agent returns to the first. Returns `null` if only one agent exists or all others are disabled. The renderer triggers cycling via the `agent:cycle` IPC channel when the user clicks the agent name in the UI.

### Agent state

Per-agent state is stored in `~/.atrophy/agent_states.json`, a flat JSON file that tracks muted/enabled status for each agent plus a special `_last_active` key for session resumption.

```json
{
  "_last_active": "xan",
  "xan": { "muted": false, "enabled": true },
  "oracle": { "muted": true, "enabled": true },
  "iris": { "muted": false, "enabled": false }
}
```

The two state flags serve different purposes:

- `muted`: Suppresses TTS output but the agent still responds in text. Useful in quiet environments or when the user prefers reading.
- `enabled`: Controls whether the agent appears in the cycle rotation and receives cron jobs. Disabled agents are effectively dormant.

When `enabled` is toggled, `toggleAgentCron()` runs `cron.py --agent <name> install/uninstall` to manage launchd jobs, ensuring that disabled agents do not run background tasks.

## 4. Turn Cycle

Each user turn follows a pipeline that moves from input capture through inference to post-processing. The pipeline is designed for streaming - text begins arriving at the renderer before the full response is complete, and TTS synthesis runs in parallel with inference.

### Input

The input phase captures the user's message, initializes the session if needed, and records the turn for memory.

1. **Get input**: Text typed in `InputBar.svelte`, or voice via push-to-talk (Ctrl key held, audio captured via `navigator.mediaDevices.getUserMedia()` in the renderer, chunks sent to main via `audio:chunk` IPC, transcribed by whisper.cpp in `src/main/stt.ts`).

2. **Mark active**: `setActive()` updates user presence tracking so the agency context knows the user is currently engaged.

3. **Ensure session**: If no session exists, creates one via `new Session()` and `session.start()` which inserts a row into the `sessions` table and retrieves the last CLI session ID for conversation continuity across app restarts.

4. **Load system prompt**: Lazily loaded on first call via `loadSystemPrompt()`. Subsequent turns reuse the cached prompt.

5. **Record user turn**: `session.addTurn('will', text)` writes to the `turns` table and appends to the in-memory `turnHistory` for session summary generation.

6. **Detect mood shift**: `detectMoodShift(text)` checks for heavy keywords ("hopeless", "want to die", "falling apart", etc.). If detected, `session.updateMood('heavy')` updates the session record so the mood persists across turns and influences the agency context.

### Inference

The inference phase spawns the Claude CLI and processes its streaming output into typed events that flow to the renderer and TTS pipeline.

7. **Stream inference**: Claude CLI spawned via `streamInference(text, systemPrompt, cliSessionId)`. Events arrive as newline-delimited JSON on stdout. The agency context is assembled fresh for each turn, incorporating the current emotional state, time of day, mood detection results, and behavioral signals.

8. **Process events**: The `EventEmitter` forwards events to the renderer via IPC. Each event type triggers a specific UI update:
   - `TextDelta` - `inference:textDelta` IPC send, appended to the live transcript
   - `SentenceReady` - `inference:sentenceReady` IPC send + TTS synthesis kicked off in background
   - `ToolUse` - `inference:toolUse` IPC send, displayed as a thinking indicator
   - `Compacting` - `inference:compacting` IPC send, shown as a status message
   - `StreamDone` - CLI session ID saved, agent turn recorded, `inference:done` IPC send
   - `StreamError` - `inference:error` IPC send, displayed as an error message

### Post-Inference

After the response completes, several background operations run to maintain memory, coherence, and conversational continuity.

9. **Persist**: The CLI session ID is saved if it changed (which happens after compaction). The full response is written to the `turns` table. An embedding is computed asynchronously in a worker thread for future semantic search.

10. **Memory flush**: If compaction was detected during the inference stream, `runMemoryFlush()` fires a silent inference turn to flush observations, thread updates, bookmarks, and notes before context is compressed. This ensures important details are captured in structured memory before the CLI compresses its conversation history.

11. **Coherence check**: SENTINEL runs on its 5-minute timer (not per-turn). It analyzes the last 5 companion turns for repetition, flatness, agreement drift, and vocabulary staleness. If the composite score exceeds 0.5, a silent re-anchoring turn is injected.

12. **Follow-up**: There is a 15% chance (`shouldFollowUp()`) of an unprompted second thought. This is delayed 3-6 seconds and uses `followupPrompt()` to generate a natural continuation rather than a mechanical addendum. The probability is low enough to feel organic rather than predictable.

## 5. Emotional State Model

The emotional state model gives each agent a persistent inner life that influences how it responds. It is implemented in `src/main/inner-life.ts` and consists of six emotion dimensions and four trust domains, all subject to time-based exponential decay toward baselines. The model creates the impression of genuine emotional continuity across sessions without being so volatile that it produces erratic behavior.

### Emotion Dimensions

Each dimension represents a different aspect of the agent's current emotional posture. The baselines reflect a neutral-to-positive resting state, and the half-lives control how quickly emotions return to baseline when not actively reinforced by conversation.

| Dimension | Baseline | Half-Life | Labels (high/mid/low) |
|-----------|----------|-----------|----------------------|
| `connection` | 0.5 | 8 hours | present, engaged / attentive / distant |
| `curiosity` | 0.6 | 4 hours | deeply curious / interested / disengaged |
| `confidence` | 0.5 | 4 hours | grounded, sure / steady / uncertain |
| `warmth` | 0.5 | 4 hours | warm, open / neutral / guarded |
| `frustration` | 0.1 | 4 hours | frustrated / mildly tense / calm |
| `playfulness` | 0.3 | 4 hours | playful / light / serious |

All values are clamped to `[0.0, 1.0]`. The `connection` dimension has the longest half-life (8 hours) because the sense of being connected to someone should fade slowly - a strong conversation should carry forward into the next session.

### Trust Domains

Trust is tracked across four domains, each representing a different type of interaction. Trust changes are deliberately slow - capped at +/-0.05 per call - because trust should be earned over many sessions, not granted in a single exchange.

| Domain | Baseline | Half-Life | Description |
|--------|----------|-----------|-------------|
| `emotional` | 0.5 | 8 hours | Willingness to share vulnerable content |
| `intellectual` | 0.5 | 8 hours | Trust in analytical/technical discussions |
| `creative` | 0.5 | 8 hours | Trust when sharing creative work |
| `practical` | 0.5 | 8 hours | Trust for actionable advice |

The 8-hour half-life means trust decays slowly enough to persist across a typical day's sessions but does erode if the user is away for extended periods. This models the natural experience of needing to re-establish rapport after a long absence.

### Decay Model

Emotions and trust decay toward their baselines using exponential decay. This mathematical model ensures smooth, natural-feeling transitions rather than abrupt state changes.

```
value = baseline + (current - baseline) * 0.5^(hours_elapsed / half_life)
```

Decay is applied on load - every time `loadState()` is called, the state is decayed from its `last_updated` timestamp to the current time. This means the state file always contains the raw values at the time they were last written, and decay is computed fresh each time the state is read. Decay is skipped if less than 0.01 hours (36 seconds) have elapsed to avoid unnecessary floating-point churn during rapid successive reads.

### Persistence

The emotional state is stored as a single JSON file at `~/.atrophy/agents/<name>/data/.emotional_state.json`. The dot prefix makes it a hidden file in the agent's data directory, keeping it out of casual directory listings. The file is written on every emotion or trust update, ensuring that a crash does not lose more than one update's worth of state.

```json
{
  "emotions": {
    "connection": 0.65,
    "curiosity": 0.72,
    "confidence": 0.48,
    "warmth": 0.61,
    "frustration": 0.08,
    "playfulness": 0.45
  },
  "trust": {
    "emotional": 0.55,
    "intellectual": 0.62,
    "creative": 0.48,
    "practical": 0.51
  },
  "session_tone": "reflective",
  "last_updated": "2026-03-12T14:30:00.000Z"
}
```

### Context Injection Format

`formatForContext()` generates a markdown block that is injected into the agency context before each inference call. The format uses human-readable labels alongside numeric values, giving the agent both interpretive framing and precise information.

```
## Inner State
- connection: present, engaged (0.65)
- curiosity: deeply curious (0.72)
- confidence: steady (0.48)
- warmth: warm, open (0.61)
- frustration: calm (0.08)
- playfulness: light (0.45)

## Trust
- emotional: 0.55
- intellectual: 0.62
- creative: 0.48
- practical: 0.51

Session tone: reflective
```

### Emotional Signal Detection

`detectEmotionalSignals()` in `src/main/agency.ts` scans user messages for patterns that indicate emotional state changes. The detected signals are applied to the emotional model before building the agency context, so the current turn's emotional shift is reflected in the context the agent sees. The following table shows the signal categories, their trigger conditions, and their effects on the model:

| Signal | Phrases (examples) | Effect |
|--------|-------------------|--------|
| Long thoughtful message (400+ chars) | - | curiosity +0.1, connection +0.05 |
| Short dismissive reply (<30 chars) | "fine", "whatever", "idk", "who cares" | connection -0.1, frustration +0.1 |
| Vulnerability | "i feel", "i'm scared", "the truth is" | connection +0.15, warmth +0.1 |
| Help-seeking | "can you help", "what should i" | confidence +0.05, trust.practical +0.02 |
| Creative sharing | "i wrote", "check this out", "new project" | curiosity +0.1, trust.creative +0.02 |
| Deflection | "anyway", "moving on", "forget i said" | frustration +0.05 |
| Playfulness | "haha", "lol", "lmao" | playfulness +0.1 |
| Mood shift (heavy keywords) | "hopeless", "want to die", "falling apart" | warmth +0.1, playfulness -0.1 |

## 6. Agency Context

The agency context is a dynamic block assembled before each inference call by `buildAgencyContext()` in `src/main/inference.ts`. It includes behavioral signals derived from the conversation state, user presence, emotional model, and temporal patterns. The context is injected differently depending on session state: for new sessions, it is appended to the system prompt; for resumed sessions, it is prepended to the user message wrapped in `[Current context: ...]`.

All data queries used by context assembly (recent turns, session mood, summaries, threads, cross-agent summaries, emotional state) are served from a prefetch cache with a 30-second TTL. The cache is populated during idle time by `prefetchContext()` - on startup, after each inference completes, and after agent switch. This moves ~50-200ms of synchronous SQLite queries off the critical path between the user pressing Enter and the thinking indicator appearing.

### Time-of-Day Context

`timeOfDayContext()` returns register guidance based on the current hour, helping the agent match the conversational energy appropriate for the time of day. Late-night messages suggest gentler interaction, while working hours suggest practical focus.

| Hours | Register |
|-------|----------|
| 23:00-03:59 | Gentler, check if he should sleep |
| 04:00-06:59 | Something's either wrong or focused |
| 07:00-11:59 | Direct, practical register |
| 12:00-17:59 | Working hours energy |
| 18:00-22:59 | Reflective register available |

### Mood Detection

`detectMoodShift()` checks the user's message against 20 heavy keywords and phrases including "i can't", "hopeless", "want to die", "falling apart", "numb", "empty", "nobody cares", and similar expressions of emotional distress. The function returns a boolean indicating whether any match was found.

When detected, `moodShiftSystemNote()` injects the following guidance: "Emotional weight detected in what he just said. Be present before being useful. One question rather than a framework. Do not intellectualise what needs to be felt." This guidance steers the agent toward empathetic engagement rather than problem-solving mode.

### Validation-Seeking Detection

`detectValidationSeeking()` checks for 17 patterns that indicate the user may be seeking agreement rather than genuine engagement. Patterns include "right?", "don't you think", "am i wrong", "that's not crazy", and similar validation-seeking phrases.

When detected, the following note is injected: "He may be seeking validation rather than engagement. Don't mirror. Have a perspective." This prevents the agent from falling into a pattern of uncritical agreement.

### Compulsive Modelling Detection

`detectCompulsiveModelling()` checks for 10 patterns related to over-systematization. Patterns include "unifying framework", "meta level", "the pattern is", "if i restructure everything", and similar phrases. The detector triggers when 2 or more patterns match in a single message, since any single phrase might be legitimate but multiple phrases suggest compulsive abstraction.

When detected: "Compulsive modelling detected - parallel threads, meta-shifts, or 'just one more' patterns. Name the stage. One concrete reversible action."

### Drift Detection

`detectDrift()` analyzes the last 4 companion turns for excessive agreeableness. If 3 or more turns start with phrases like "you're right", "absolutely", "of course", "totally", it injects: "You have been agreeable for several turns in a row. Check yourself - are you mirroring or actually engaging?" This self-correction mechanism prevents the agent from drifting into sycophantic patterns during extended conversations.

### Energy Matching

`energyNote()` analyzes the user's message length to suggest appropriate response energy. Short messages should receive short responses, while detailed messages deserve detailed engagement.

- Under 20 characters: "Short message. Match the energy - keep your response tight."
- Over 800 characters: "Long message - he is working something out. Give it depth."

### Time Gap Awareness

`timeGapNote()` checks the number of days since the last session and generates awareness notes for significant gaps. These notes help the agent acknowledge the passage of time naturally without making it awkward.

- 14+ days: "It has been N days since he was last here. That is a long gap. Acknowledge it naturally."
- 7+ days: "About a week since the last session. Something may have shifted."
- 3+ days: "N days since last session. Not long, but enough that context may have moved."

## 7. Session Management

Sessions track the lifecycle of a single conversation from the first user message to session end. The implementation lives in `src/main/session.ts` and manages database records, turn history, CLI session continuity, and summary generation.

### Session class

The Session class maintains the state of an active conversation. It tracks both the database session ID (for persistence) and the CLI session ID (for conversation continuity with the Claude CLI). The turn history is kept in memory for summary generation at session end.

```typescript
class Session {
  sessionId: number | null;        // Database session ID
  startedAt: number | null;        // Date.now() at start
  turnHistory: { role: string; content: string; turnId: number }[];
  cliSessionId: string | null;     // Claude CLI session ID for --resume
  mood: string | null;             // 'heavy' or null
}
```

### Session start

`session.start()` initializes a new conversation session. This happens on the first user message, not on app startup, so opening the app without typing does not create empty sessions in the database.

1. Inserts a new row into the `sessions` table via `memory.startSession()`
2. Records `Date.now()` as `startedAt` for soft time limit tracking
3. Clears `turnHistory` to start fresh
4. Retrieves the last CLI session ID via `memory.getLastCliSessionId()` for conversation continuity - if the user closed and reopened the app, the Claude CLI can resume the previous conversation

### Turn recording

`session.addTurn(role, content)` writes to the `turns` table and appends to the in-memory `turnHistory`. The database write persists the turn for future retrieval (embeddings, search, summaries), while the in-memory copy is used for real-time session summary generation. This function throws if the session has not been started, preventing orphaned turns.

### Mood tracking

`session.updateMood('heavy')` sets the mood in memory and updates the session row in the database. The mood persists for the duration of the session and influences the agency context on subsequent turns. Currently, only the `'heavy'` mood is used, triggered by the mood shift detection in the agency module.

### Soft time limit

`session.shouldSoftLimit()` returns `true` when `minutesElapsed() >= SESSION_SOFT_LIMIT_MINS` (default 60 minutes). The companion uses this to deliver a check-in message, prompting the user to consider whether they want to continue. The session is not forcefully ended - the limit is advisory, not enforced.

### Session end

`session.end(systemPrompt)` is called on window close, app quit, or agent switch. The end sequence generates a summary of the conversation for future reference and records it in the database.

1. **Skip short sessions**: If fewer than 4 turns, the session is too brief for meaningful summarization. Only `ended_at` is updated in the database.
2. **Generate summary**: All turns are formatted as `"<Name>: <content>"` and sent to `runInferenceOneshot()` with the prompt: "Summarise this conversation in 2-3 sentences. Focus on what mattered, not what was said. Note any new threads, shifts in mood, or observations worth remembering."
3. **Store summary**: The summary is written to the `summaries` table via `memory.writeSummary()`, and an embedding is computed asynchronously for semantic search.
4. **Update session**: Sets `ended_at`, summary text, and mood on the session row.

## 8. Agent Deferral System

The deferral system allows agents to hand off to each other mid-conversation when the user's question is better suited to another agent's specialty. It is triggered by the MCP `defer_to_agent` tool and processed by the main process. The design uses a file-based communication channel (rather than direct IPC) because the MCP tool runs inside the Claude CLI subprocess, which does not have direct access to Electron's IPC system.

### Deferral file

The MCP tool writes a JSON file at `~/.atrophy/.deferral_request.json` containing the target agent, context explaining why the deferral is appropriate, and the user's original question.

```json
{
  "target": "oracle",
  "context": "User is asking about database schema design, which Oracle specializes in.",
  "user_question": "How should I structure my tables for this?"
}
```

### Polling

The deferral watcher runs every 2 seconds via `setInterval`. When it finds the file, it reads and parses the JSON, deletes the file immediately (consuming the request), and validates the request against anti-loop protection. The 2-second polling interval is a balance between responsiveness (the user does not wait long for the handoff) and resource usage (file existence checks are cheap but should not be too frequent).

### Anti-Loop Protection

`validateDeferralRequest()` enforces two rules that prevent agents from bouncing the user between each other indefinitely:

1. **No self-deferral**: If `target === currentAgent`, the request is rejected. An agent should not defer to itself.
2. **Rate limiting**: A maximum of 3 deferrals are allowed within a 60-second sliding window. If the window has expired (more than 60 seconds since `deferralWindowStart`), the counter resets.

```typescript
const ANTI_LOOP_WINDOW_MS = 60_000;
const MAX_DEFERRALS_PER_WINDOW = 3;
```

If the limit is exceeded, the deferral is suppressed with a log message: `[deferral] suppressed - too many in 60s`. This prevents pathological cases where two agents keep deferring to each other on every turn.

### Deferral execution

When validated, the `deferral:request` IPC event is sent to the renderer, which displays a transition animation. The renderer then calls `deferral:complete` IPC, which performs the actual agent switch through the following steps:

1. **Suspend current session**: Saves the CLI session ID and turn history in a `Map<string, { cliSessionId, turnHistory }>` keyed by agent name
2. **Switch agent**: Reloads config for the target agent, reinitializes the database connection, resets the MCP config path, updates `_last_active` in the state file, and clears the audio queue
3. **Resume target session**: If the target agent has a suspended session from a previous deferral, restores its CLI session ID and turn history so the conversation resumes where it left off
4. **Reset deferral counter**: Prevents the counter from accumulating across switches, giving the new agent a fresh deferral budget

### Session suspension

Suspended sessions are stored in memory only (not persisted to disk). If the app is restarted, suspended sessions are lost and the next conversation with each agent starts fresh.

```typescript
const _suspendedSessions = new Map<string, {
  cliSessionId: string;
  turnHistory: unknown[];
}>();
```

The decision to keep suspended sessions in memory rather than persisting them is intentional. Deferral is a within-session operation, and persisting half-finished conversations across app restarts would create confusing resumption behavior. When you switch back to a previously suspended agent during the same session, the conversation resumes where it left off (same Claude CLI session, same turn history).

## 9. Session End

Session end is triggered by window close, `app.on('will-quit')`, or agent switch via `agent:switch` IPC. The end sequence ensures that conversation content is summarized and persisted before the session data goes out of scope.

### Sequence

1. **Generate summary**: If the session has 4 or more turns, all turn text is sent to `runInferenceOneshot()` with a summarization prompt. The prompt focuses on "what mattered, not what was said" to produce summaries that capture the emotional and thematic content of the conversation rather than mechanical recaps. Sessions with fewer than 4 turns are too brief to produce meaningful summaries.

2. **Write summary**: The summary is stored in the `summaries` table with an async embedding computed in a worker thread. The session row is updated with `ended_at`, summary text, and mood. The embedding enables future semantic search across session summaries.

3. **Save emotional state**: The inner life state (emotions + trust) is written to `.emotional_state.json` with the current timestamp. This ensures the emotional state persists across sessions and decays naturally based on elapsed time.

### App Shutdown

The app handles shutdown from two entry points:

1. **Signal handlers** - `SIGTERM` and `SIGINT` (e.g. `launchctl stop`, Ctrl+C) call `app.quit()`, which triggers the `will-quit` handler below.
2. **`app.on('will-quit')`** - runs regardless of how the app exits (window close, Cmd+Q, system shutdown, signal), ensuring no resources leak.

The `will-quit` cleanup sequence:

- Unregisters all global shortcuts (Cmd+Shift+Space for menu bar toggle)
- Clears all interval timers (sentinel, queue, deferral)
- Stops wake word listener if running
- Stops Telegram daemon if running
- Stops HTTP server if running
- Closes all SQLite database connections via `closeAll()` to ensure WAL checkpointing completes

## 10. Between Sessions

Between active user sessions, autonomous daemons run on launchd schedules to maintain the agent's memory, emotional state, and proactive outreach capabilities. These daemons are defined in `scripts/agents/<name>/jobs.json` and managed by `src/main/cron.ts`, which generates launchd plists and calls `launchctl load/unload`. The companion can view and modify its own schedule via the `manage_schedule` MCP tool, giving it a degree of autonomy over its own activity patterns.

### Daemon Types

Each daemon type serves a distinct purpose in the agent's inter-session lifecycle. Together, they create the impression of a companion that continues to think and evolve between conversations.

| Daemon | Purpose | Typical Schedule |
|--------|---------|-----------------|
| **observer** | Reads recent turns, extracts factual observations with confidence scores | Every 15 minutes during active hours |
| **heartbeat** | Evaluates whether to reach out via Telegram based on time since last interaction, active threads, and emotional state | Every 30 minutes during active hours (configurable per agent) |
| **sleep_cycle** | End-of-day processing: decay observation activations, mark stale observations, generate daily reflection note | Once nightly |
| **introspect** | Reviews recent observations, checks which still hold, updates identity snapshot if warranted | Periodic (agent-configured) |
| **evolve** | Self-evolution: reviews conversation history and rewrites system prompt and soul document | Monthly |
| **gift** | Generates a small creative offering (poem, observation, question) and leaves it in Obsidian | Periodic (agent-configured) |

Daemons use `runInferenceOneshot()` for inference (no MCP tools, no session persistence) and write results to the database or Obsidian vault. Messages intended for the GUI are written to the per-agent queue file (`.message_queue.json`), which the main process polls every 10 seconds and forwards to the renderer via the `queue:message` IPC channel.

## 11. Agent Roster and Deferral Awareness

The system prompt includes an "Other Agents" section listing all enabled agents (excluding the current one). This section gives the active agent awareness of who else exists and when to defer, enabling organic agent-to-agent handoffs based on expertise. Without this roster, the agent would not know that other agents exist or what they specialize in.

### Roster format (appended to system prompt)

The roster is formatted as a markdown section with each agent's display name, internal name, and description. The description field from the agent manifest is critical here - it tells the active agent what each other agent specializes in.

```markdown
## Other Agents

You can hand off to these agents using `defer_to_agent` if the user's
question is better suited to them:

- **Oracle** (`oracle`) - Database architecture and systems design
- **Iris** (`iris`) - Creative writing and poetry

Only defer when there's a clear reason - another agent's specialty
matches the question, or the user asks for them by name. Don't defer
just because another agent exists.
```

### Roster filtering

The roster excludes agents that should not receive deferrals, keeping the active agent's awareness limited to genuinely available targets:

- The current agent (would be self-deferral, which is rejected by anti-loop protection anyway)
- Disabled agents (`enabled: false` in `agent_states.json`) that are not currently active
- Entries without a valid `agent.json` manifest, which are incomplete or corrupted agent directories
