# API Guide

The Atrophied Mind runs as a headless HTTP server for remote access, web frontends, and programmatic integration. No GUI, no TTS, no voice input - just a REST API over localhost.

---

## Starting the Server

The server runs when the `--server` flag is passed to the Electron app:

```bash
pnpm dev -- --server                    # localhost:5000 (development)
pnpm dev -- --server --port 8080        # custom port
AGENT=oracle pnpm dev -- --server       # run a specific agent
```

In a packaged `.app`, pass the flags via the command line or configure the launch arguments.

On startup the server prints a summary including a truncated token, the token file path, and all available endpoints.

The `AGENT` environment variable controls which agent is loaded (default: `xan`). The agent's system prompt, memory database, and identity are all resolved from that name.

### Binding to a custom host

By default the server binds to `127.0.0.1` (localhost only). To expose it on all network interfaces - for example, to access the API from another machine on your LAN - pass the `--host` flag:

```bash
pnpm dev -- --server --host 0.0.0.0              # All interfaces, default port
pnpm dev -- --server --host 0.0.0.0 --port 8080  # All interfaces, custom port
```

The `startServer()` function in `src/main/server.ts` accepts an optional `host` parameter (default `'127.0.0.1'`). Bearer token auth still applies on all endpoints except `/health`, but binding to `0.0.0.0` exposes the API to your entire network - use with caution.

## Authentication

All endpoints except `/health` require a Bearer token.

### Where the token lives

The token is auto-generated on first run and stored at:

```
~/.atrophy/server_token
```

The file is created with `0600` permissions (owner read/write only). The token is a 32-byte URL-safe random string generated by `crypto.randomBytes(32).toString('base64url')`.

### How to retrieve it

```bash
cat ~/.atrophy/server_token
```

### How to pass it

Include the token in the `Authorization` header on every request:

```
Authorization: Bearer <token>
```

If the token is missing or wrong, every protected endpoint returns `401`:

```json
{"error": "unauthorized"}
```

## Endpoints

### GET /health

Health check. No auth required.

**Response:**

```json
{
  "status": "ok",
  "agent": "xan",
  "display_name": "Xan"
}
```

**Example:**

```bash
curl http://localhost:5000/health
```

---

### POST /chat

Send a message, receive the full response when inference completes. Blocks until the agent finishes generating.

**Request body:**

```json
{
  "message": "What have we been talking about lately?"
}
```

**Response:**

```json
{
  "response": "We've been discussing...",
  "session_id": "abc123"
}
```

**Error (empty message):** `400`

```json
{"error": "empty message"}
```

**Error (inference in progress):** `429`

```json
{"error": "inference in progress"}
```

**Error (inference failure):** `500`

```json
{"error": "inference failed: ..."}
```

**Example:**

```bash
TOKEN=$(cat ~/.atrophy/server_token)

curl -X POST http://localhost:5000/chat \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"message": "How are you?"}'
```

Note: the `/chat` endpoint holds a mutex for the duration of inference. Concurrent requests receive a `429` response rather than queuing. This is intentional - the agent maintains conversational state and turn order matters.

---

### POST /chat/stream

Send a message, receive a streaming response via Server-Sent Events (SSE). Tokens arrive incrementally as the agent generates them.

**Request body:**

```json
{
  "message": "Tell me something interesting."
}
```

**Response:** `text/event-stream`

Each SSE frame is a `data:` line containing a JSON object with a `type` field:

| Type | Fields | Meaning |
|------|--------|---------|
| `text` | `content` | A chunk of generated text |
| `tool` | `name` | The agent invoked an MCP tool |
| `error` | `message` | An error occurred during inference |
| `done` | `full_text` | Generation complete; contains the full assembled response |

**Example SSE stream:**

```
data: {"type": "text", "content": "The"}

data: {"type": "text", "content": " interesting thing"}

data: {"type": "text", "content": " about memory is..."}

data: {"type": "tool", "name": "remember"}

data: {"type": "text", "content": " I recall that you mentioned..."}

data: {"type": "done", "full_text": "The interesting thing about memory is... I recall that you mentioned..."}
```

**Example:**

```bash
TOKEN=$(cat ~/.atrophy/server_token)

curl -N -X POST http://localhost:5000/chat/stream \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"message": "Tell me something interesting."}'
```

The `-N` flag disables curl's output buffering so SSE events print as they arrive.

### GET /memory/search

Search the agent's memory using hybrid vector + keyword search.

**Query parameters:**

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `q` | Yes | - | Search query |
| `limit` | No | `10` | Max results to return |

**Response:**

```json
{
  "results": [
    {
      "content": "Will mentioned he's been working on...",
      "score": 0.82,
      "source": "turns",
      "timestamp": "2026-03-09T14:30:00"
    }
  ]
}
```

**Error (missing query):** `400`

```json
{"error": "missing q parameter"}
```

**Example:**

```bash
TOKEN=$(cat ~/.atrophy/server_token)

curl "http://localhost:5000/memory/search?q=project+deadline&limit=5" \
  -H "Authorization: Bearer $TOKEN"
```

## Error Handling

All errors return JSON with an `error` field.

| Status | Meaning | Example |
|--------|---------|---------|
| `400` | Bad request - missing or empty required field | `{"error": "empty message"}` |
| `401` | Unauthorized - missing or invalid Bearer token | `{"error": "unauthorized"}` |
| `404` | Not found - unknown endpoint | `{"error": "not found"}` |
| `429` | Too many requests - inference already in progress | `{"error": "inference in progress"}` |
| `500` | Server error - inference failure or internal exception | `{"error": "internal server error"}` |

Streaming errors (`/chat/stream`) are delivered as SSE events with `type: "error"` rather than HTTP status codes, since the response has already started with a `200`.

## Integration Examples

### TypeScript / Node.js

```typescript
import { readFileSync } from 'fs';
import { join } from 'path';
import { homedir } from 'os';

const BASE = 'http://localhost:5000';
const TOKEN = readFileSync(join(homedir(), '.atrophy', 'server_token'), 'utf-8').trim();
const HEADERS = {
  Authorization: `Bearer ${TOKEN}`,
  'Content-Type': 'application/json',
};

// Simple chat
const resp = await fetch(`${BASE}/chat`, {
  method: 'POST',
  headers: HEADERS,
  body: JSON.stringify({ message: 'Hello' }),
});
const data = await resp.json();
console.log(data.response);

// Streaming chat
const stream = await fetch(`${BASE}/chat/stream`, {
  method: 'POST',
  headers: HEADERS,
  body: JSON.stringify({ message: 'Tell me about my week' }),
});
const reader = stream.body!.getReader();
const decoder = new TextDecoder();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  const text = decoder.decode(value);
  for (const line of text.split('\n')) {
    if (line.startsWith('data: ')) {
      const event = JSON.parse(line.slice(6));
      if (event.type === 'text') process.stdout.write(event.content);
      if (event.type === 'done') console.log();
    }
  }
}

// Memory search
const searchResp = await fetch(
  `${BASE}/memory/search?q=project+deadline&limit=5`,
  { headers: HEADERS },
);
const results = await searchResp.json();
for (const r of results.results) {
  console.log(r);
}
```

### Python (requests)

```python
import json
from pathlib import Path

import requests

BASE = "http://localhost:5000"
TOKEN = (Path.home() / ".atrophy" / "server_token").read_text().strip()
HEADERS = {
    "Authorization": f"Bearer {TOKEN}",
    "Content-Type": "application/json",
}

# Simple chat
resp = requests.post(f"{BASE}/chat", headers=HEADERS, json={"message": "Hello"})
data = resp.json()
print(data["response"])

# Streaming chat
resp = requests.post(
    f"{BASE}/chat/stream",
    headers=HEADERS,
    json={"message": "Tell me about my week"},
    stream=True,
)
for line in resp.iter_lines(decode_unicode=True):
    if line.startswith("data: "):
        event = json.loads(line[6:])
        if event["type"] == "text":
            print(event["content"], end="", flush=True)
        elif event["type"] == "done":
            print()

# Memory search
resp = requests.get(
    f"{BASE}/memory/search",
    headers=HEADERS,
    params={"q": "project deadline", "limit": 5},
)
for r in resp.json()["results"]:
    print(r)
```

### JavaScript (EventSource - GET only)

The `/chat/stream` endpoint uses POST, so the native `EventSource` API (which only supports GET) cannot be used directly. Use `fetch` with a streaming reader as shown above, or a library like `eventsource-parser`.

## Configuration

### Relevant environment variables

| Variable | Default | Effect on server mode |
|----------|---------|----------------------|
| `AGENT` | `xan` | Which agent to load |
| `ATROPHY_DATA` | `~/.atrophy` | Root for runtime data (token file, memory DBs) |
| `CLAUDE_BIN` | `claude` | Path to Claude CLI binary used for inference |
| `CLAUDE_EFFORT` | `medium` | Inference effort level |
| `ADAPTIVE_EFFORT` | `true` | Auto-adjust effort by query complexity |

### How server mode differs from GUI/App

| Feature | GUI/App | Server |
|---------|---------|--------|
| Voice input | Yes | No |
| TTS output | Yes | No |
| Avatar | Yes | No |
| Wake words | Yes | No |
| Heartbeat/cron | Yes (via separate launchd jobs) | Same (independent of mode) |
| Memory | Full | Full |
| MCP tools | Full | Full |
| Session tracking | Full | Full |
| Concurrency | Single-threaded | Single-threaded (mutex) |

The server uses Node's built-in `http` module running in the Electron main process. For production use behind a reverse proxy, bind to `127.0.0.1` (the default) and proxy from nginx or similar.

---

## Concurrency

The server holds a boolean mutex (`inferLock`) during inference. This means:

- Only one `/chat` or `/chat/stream` request is processed at a time.
- Additional requests receive a `429` response immediately rather than queuing.
- Memory and session endpoints are not locked and respond immediately.

This is by design. The agent maintains conversational state and turn ordering must be preserved.
